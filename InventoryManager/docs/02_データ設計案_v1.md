## 設計方針

- **台帳(イベントログ)方式**: 在庫の増減・状態変化・移動をすべて 1 行 = 1 イベントとして記録する。
- **現在庫は台帳から集計して導出する**(スナップショットを別途持たない)。規模が小さい(品番 60 × 最大 30 個 × 週次)ため、SUMIFS/ピボットで都度計算して十分。
- データは UI(VBA / PowerShell+XAML)から独立した「ただのテーブル(シート)」として設計する。
- 入力ミスは行の直接修正・削除で訂正する(取消行方式は採らない)。

## 在庫の粒度

倉庫Bの在庫は次の 4 軸の内訳で把握する:

```
品番 × ロット(または未割当) × 処理段階(状態) × 数量
```

- **ロットID = 品番 + 出荷週(月曜日付)** で一意。例: `ABC-123_2026-07-13`
- **納期先はロットキーに含めない**。同一品番・同一週で納期先が複数ある場合(A に 5、B に 6)も、在庫・工程管理は合算 11 個の 1 ロットとして扱う(充足判定も合算で良い ※確定)。納期先は納期計画の内訳と、梱包以降の記録の属性としてのみ持つ。
- ロットに紐付かない在庫は「**未割当(余剰)**」として品番単位で持つ。
- 状態: `未処理` / `アニール済み` / `アセンブリ済み` / `梱包済み` / `発送準備完了` / `不良`

| 品名 | |

## テーブル 2: T_納期計画

納期一覧表 Excel から取り込んで作る。**1 行 = 品番 × 出荷週 × 納期先**(納期先が分かれる場合は複数行になる)。

| 列 | 内容 |
|---|---|
| ロットID | 品番 + "_" + 出荷週月曜日(グループキー。納期先違いの複数行が同じロットIDを持つ) |
| 品番 | |
| 出荷週 | 月曜日の日付 |
| 納期先 | 出荷先(1 つしかない場合は空欄でも可) |
| 必要数 | 納期一覧表のセル値(納期先ごと) |
| 計画状態 | 有効 / キャンセル(再取込でセルが空になったらキャンセル扱い) |
| 最終取込日時 | |

- 主キーは 品番 × 出荷週 × 納期先。
- 充足判定はロットID(品番×週)で **必要数を合算** して行う。

### 取込(再取込)の動き

納期変更・数量変更・キャンセルが「ちょくちょく」あるため、**取込は毎回全量スキャン+突合**とする:

1. 納期一覧表の 2 行目(E 列以降)から週の月曜日付を読む。
2. 6 行目以降の品番行 × 週列の交点で、数量が入っているセルを列挙する。
3. T_納期計画と突合:
   - 新しい 品番×週 → 行を追加
   - 数量が変わった → 必要数を更新
   - 表から消えた(セルが空になった) → 計画状態をキャンセルに変更(行は消さない。既に搬入済みの実績と突き合わせるため)

## テーブル 3: T_台帳(イベントログ)— 中核

1 行 = 1 イベント。

| 列 | 内容 |
|---|---|
| No | 連番 |
| 実施日 | 作業を行った日 |
| イベント種類 | 下表参照 |
| 品番 | |
| ロットID | 対象ロット。未割当在庫が対象なら空欄 |
| 充当先ロットID | 「充当」イベントのみ使用 |
| 元状態 | どの状態の在庫を対象にしたか(部分進行があるため、**搬入以外の全イベントで必須**) |
| 数量 | |
| 納期先 | 任意。梱包完了/発送準備完了/発送で納期先別の進捗を残したい場合に記入 |
| 記録者 / 備考 | |

### イベント種類と在庫への影響

| イベント種類 | 在庫への影響(倉庫B) | 先状態 |
|---|---|---|
| 搬入(A→B) | ロットに +N | 未処理 |
| アニール完了 | 元状態の在庫を N 個、状態変更 | アニール済み |
| アセンブリ完了 | 同上 | アセンブリ済み |
| 梱包完了 | 同上 | 梱包済み |
| 発送準備完了 | 同上 | 発送準備完了 |
| 発送 | ロットから −N(在庫から消える) | — |
| 不良発生 | 元状態の在庫を N 個、不良へ | 不良 |
| 廃棄 | 不良在庫から −N | — |
| 倉庫A戻し(B→A) | 対象在庫から −N(状態・元ロットは備考/元状態で残る) | — |
| 余剰化 | ロット → 未割当へ N 個振替(状態維持) | (維持) |
| 充当 | 未割当(またはロット)→ 充当先ロットへ N 個振替(状態維持) | (維持) |

- 「先状態」はイベント種類から一意に決まるため、列としては持たない。
- 元状態は状態変更イベントに限らず、**在庫を減らす/移すイベント(発送・廃棄・倉庫A戻し・余剰化・充当)でも記録する**。これにより「ロット×状態」別の在庫内訳が台帳だけから機械的に集計できる(廃棄の元状態は常に「不良」、発送は通常「発送準備完了」なので UI 側で自動補完すれば入力負担はない)。
- 状態遷移の妥当性(例: 梱包完了の元状態は、その品番の工程パターンの最終処理段階であること)は入力時に UI 側でチェックする。

## 導出ビュー(集計)

台帳から次を計算して表示する:

### V_在庫内訳(優先度 2: ステータス把握)

`品番 × ロット/未割当 × 状態` ごとの現在数量。台帳の全イベントを上表のルールで加減算した結果。

### V_充足確認(優先度 1)

有効な各ロット(T_納期計画)について:

```
必要数合計 = 同一ロットID(品番×週)の必要数の合算(納期先違いを合算)
充足数   = ロットに紐付く現在庫(不良を除く全状態)
         + 同品番の未割当在庫(不良を除く)
不足数   = 必要数合計 − 充足数(正なら「倉庫Aから搬出が必要」)
```

- 別ロットに割当済みの在庫はカウントしない(確定ルール)。
- 未割当在庫は同品番の複数ロットの充足判定に重複してカウントされ得る点に注意(表示上「未割当を含む」と分かるようにする)。実際にどのロットに使うかは「充当」イベントで確定させる。

## シート構成(Excel ブック「在庫管理台帳.xlsx」として実装済み)

| シート | 内容 | テーブル名 |
|---|---|---|
| 説明 | 入力ルール・クイックガイド(これをしたい→どこに入力) | — |
| 業務フロー | 状態(箱)とイベント(矢印)の流れ図(オートシェイプ) | — |
| V_充足確認 | 充足判定(数式・自動計算) | — |
| V_在庫内訳 | 状態別在庫の集計(数式・自動計算) | — |
| T_台帳 | イベントログ — 基本はここだけ日々入力 | tbl_ledger |
| T_納期計画 | 取込結果 | tbl_plan |
| M_品番 | 品番マスタ | tbl_master |
| M_リスト | 入力規則用リスト | — |

- シート名接頭辞: **V_ = View(自動計算・参照専用)/ T_ = Transaction(日々の入力データ)/ M_ = Master(基礎情報)**。
- テーブル名は `tbl_` 接頭辞+小文字始まりの命名(tbl_ledger 等)。
- **背景が灰色のセルは自動計算列**(V シート全体と T_納期計画のロットID列)。手入力しない。
